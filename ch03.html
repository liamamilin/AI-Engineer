5  Chapter 3 — Thinking in Systems

5.1 Introduction

For much of the history of software engineering, programs were understood as collections of functions. A developer defined procedures that transformed inputs into outputs through deterministic logic. If the functions were correct and the program structure was sound, the system would behave predictably.

This mental model is no longer sufficient for understanding modern AI applications.

In traditional software, a function encapsulates a transformation. Its internal logic determines the outcome. Engineers reason about behavior by examining the code itself. The correctness of the system can therefore be traced directly to the correctness of its implementation.

AI systems operate differently.

When an AI model is introduced into a program, the transformation between input and output is no longer fully determined by explicit logic. Instead, the output emerges from probabilistic inference within a trained model. The engineer cannot inspect a set of deterministic rules to explain every decision the system makes.

As a consequence, the basic unit of reasoning in AI engineering shifts.

AI is no longer best understood as a function.

AI must be understood as a system.

5.2 From Functions to Systems

5.2.1 The Function-Centric View

Traditional software engineering operates with a function-centric mental model:

f(x) → y

In this view:

  • Each function is a self-contained transformation
  • Behavior is determined by the implementation
  • Correctness is verified through testing
  • Systems are composed by calling functions

This model works exceptionally well for deterministic systems. When a function produces incorrect output, the engineer examines its implementation, identifies the bug, and corrects the logic. The relationship between cause and effect is direct and traceable.

5.2.2 The System-Centric View

AI systems require a different perspective. Instead of viewing the system as a collection of independent functions, we must view it as an integrated system whose behavior emerges from the interaction of multiple components:

System: Components → Interactions → Emergent Behavior

In this view:

  • Behavior emerges from interactions, not isolated components
  • Information flows through stages, not single transformations
  • Decisions may be revised over multiple iterations
  • Reliability emerges from architecture, not individual correctness

The shift from functions to systems is not merely conceptual—it has practical implications for how we design, build, and debug AI applications.

5.3 The Fundamental Structures

5.3.1 Pipelines: Structuring Information Flow

The most fundamental structure within AI systems is the pipeline.

A pipeline organizes a sequence of transformations that gradually refine information as it moves through the system. Each stage performs a specific operation, often combining deterministic processing with model-based inference.

A conceptual representation of a typical AI pipeline:

input
↓
preprocessing
↓
retrieval
↓
model reasoning
↓
post-processing
↓
output

Why Pipelines Matter

Pipelines introduce structure into systems that would otherwise be difficult to reason about. They make the flow of information explicit and allow engineers to:

  • Isolate individual stages for testing and debugging
  • Modify or replace components without redesigning the entire system
  • Understand how information transforms as it passes through the system
  • Compose complex systems from simpler, reusable pipelines

Pipeline Stages in Detail

Let us examine a typical pipeline more closely:

  1. Input: Raw user input or data from external sources
  2. Preprocessing: Validation, normalization, formatting, or transformation
  3. Retrieval: Fetching relevant context from external sources (documents, databases, APIs)
  4. Model Reasoning: The core inference step where the model processes the prompt and context
  5. Post-processing: Validation, formatting, filtering, or enrichment of the model output
  6. Output: The final result returned to the user or passed to another system

Each stage is independently testable and modifiable. This modularity is essential for building maintainable systems.

5.3.2 Feedback Loops: Enabling Adaptive Behavior

Pipelines alone do not fully describe modern AI systems. Many systems include feedback loops that allow the system to refine its behavior over time.

A feedback loop occurs when the output of a process influences future decisions within the system. In AI engineering, feedback may arise through:

  • Evaluation frameworks: Systems that assess output quality and feed results back to improve future behavior
  • User interactions: User feedback that influences how the system responds to similar inputs
  • Iterative reasoning: Agents that generate, evaluate, and revise their own outputs

A simple feedback loop structure:

generate
↓
evaluate
↓
revise
↓
generate

The Power of Feedback

Feedback loops transform static pipelines into adaptive systems. Instead of producing a single output, the system can iteratively improve its results based on internal or external signals.

Consider an agent solving a complex problem:

  1. Generate: The agent produces an initial solution
  2. Evaluate: The solution is assessed against criteria or tested
  3. Revise: Based on evaluation, the agent refines the solution
  4. Repeat: The cycle continues until a satisfactory result is achieved

This pattern enables capabilities that single-pass systems cannot achieve: self-correction, exploration of alternatives, and optimization of results.

5.3.3 Observability: Making Internal Behavior Visible

As AI systems become more complex, another engineering concern becomes critical: observability.

Observability refers to the ability to inspect and understand the internal behavior of a system while it is running. In traditional software systems, debugging often involves examining stack traces, inspecting variables, or reproducing errors in controlled environments.

With AI systems, observability must extend beyond code execution to include:

  • Model inputs and prompts
  • Intermediate outputs at each pipeline stage
  • Tool invocations and their results
  • Evaluation metrics and scores
  • Agent decisions and reasoning paths

Why Observability Matters

Without observability, AI systems quickly become opaque. Engineers cannot determine why a system produced a particular result or identify where failures originate.

Effective AI engineering requires:

  • Structured logging: Recording key events, decisions, and outputs throughout the system
  • Traceability: Tracking how a request flows through pipelines and agents
  • Metrics collection: Measuring performance, latency, and resource usage
  • Inspection tools: Interfaces for examining system state and behavior

Observability in Practice

Consider a question-answering system that produces an incorrect answer. With observability, the engineer can examine:

  • What prompt was constructed and sent to the model?
  • What documents were retrieved and included in context?
  • How did the model reason about the question?
  • What tools were invoked and what did they return?
  • Where in the pipeline did the error occur?

This level of visibility transforms debugging from speculation to systematic investigation.

5.3.4 Reliability: Maintaining Stable Performance Under Uncertainty

Observability enables the final essential property of robust AI systems: reliability.

Reliability in AI systems cannot be achieved by eliminating uncertainty. Probabilistic models inherently produce variation in their outputs. Instead, reliability emerges from system design.

Architecture-Based Reliability

Engineers introduce multiple mechanisms to ensure reliable behavior:

  • Evaluation frameworks: Systems that assess output quality before delivery
  • Guardrails: Constraints that prevent the system from producing unacceptable outputs
  • Fallback strategies: Alternative approaches when primary methods fail
  • Validation stages: Checks that verify outputs meet requirements
  • Retry mechanisms: Automatic retry with modified inputs when errors occur

Example: A Reliable Answer System

Consider a system designed to answer factual questions reliably:

  1. Input validation: Ensure the question is properly formatted and understandable
  2. Context retrieval: Fetch relevant documents from trusted sources
  3. Model reasoning: Generate an answer based on retrieved context
  4. Fact verification: Cross-check the answer against the source documents
  5. Confidence assessment: Determine how confident the system is in the answer
  6. Fallback: If confidence is low, either request clarification or return a conservative response

In this design, reliability is achieved through architectural layers, not through perfect model accuracy.

5.4 The System Engineer’s Mental Model

The transition from functions to systems therefore reshapes how engineers approach intelligent software.

5.4.1 Traditional Engineering

The traditional engineer thinks in terms of:

  • Algorithms and data structures
  • Functions and modules
  • Control flow and state management
  • Correctness proofs and unit tests

5.4.2 AI Engineering

The AI engineer thinks in terms of:

  • Pipelines and information flows
  • Component interactions and interfaces
  • Feedback loops and adaptive behavior
  • Observability and debugging strategies
  • Reliability through architectural design

5.4.3 The Shift in Practice

This shift changes daily engineering work:

Instead of writing a function to solve a problem directly,

The AI engineer designs a pipeline that guides a model through a sequence of transformations.

Instead of assuming that correct code guarantees correct behavior,

The AI engineer builds evaluation frameworks that continuously assess and improve system performance.

Instead of debugging by examining a single function,

The AI engineer traces how information flows through multiple stages, identifies where the transformation deviated from expectations, and adjusts the architecture accordingly.

5.5 Building System Thinking Skills

Developing the ability to think in systems requires practice. Several strategies can help engineers cultivate this mindset:

5.5.1 Visualize Information Flow

Draw diagrams showing how information moves through the system. Identify where transformations occur, where decisions are made, and where failures could occur. Visualization makes complex systems more tractable and reveals design flaws that might be invisible in code.

5.5.2 Design for Observability

From the beginning, design systems with observability in mind. Ask: How will we understand what happened? What signals do we need to collect? Where should we insert logging and metrics? Thinking about observability during design prevents opaque systems that are difficult to debug.

5.5.3 Test at Multiple Levels

Test not just the final output, but intermediate stages as well. Verify that preprocessing produces correctly formatted input. Ensure retrieval returns relevant context. Confirm that evaluation metrics accurately measure output quality. Testing at multiple levels isolates problems and provides deeper insight.

5.5.4 Embrace Iteration

System design is iterative. Start with a simple pipeline, measure its behavior, identify weaknesses, and gradually add sophistication. Each iteration should improve observability, reliability, or capability without overcomplicating the system.

5.5.5 Study Existing Architectures

Learn from well-designed systems. Examine how they structure pipelines, implement feedback loops, ensure observability, and achieve reliability. Reverse-engineering successful architectures builds intuition for what makes systems effective.

5.6 The Broader Implications

The shift to systems thinking has implications beyond individual engineering practice.

5.6.1 For Teams

Teams working on AI systems must develop shared mental models of system architecture. Cross-functional understanding—how models, pipelines, infrastructure, and evaluation interact—becomes essential. Communication must focus on system-level concerns, not just component-level details.

5.6.2 For Organizations

Organizations must invest in the infrastructure and tooling that support systems thinking: observability platforms, evaluation frameworks, and development environments designed for complex systems. They must also cultivate engineering cultures that value architectural design alongside implementation skills.

5.6.3 For Education

Engineering education must evolve to teach systems thinking alongside traditional programming skills. Students must learn to design pipelines, understand feedback mechanisms, implement observability, and reason about reliability at the system level.

5.7 Conclusion

The transition from functions to systems lies at the heart of AI engineering.

Traditional software engineering was built on the foundation of deterministic functions. AI engineering is built on the foundation of systems whose behavior emerges from the interaction of multiple components.

To build intelligent applications, engineers must move beyond the mindset of programming individual transformations. They must instead design architectures in which models, tools, data, and evaluation mechanisms interact to produce reliable outcomes.

In such architectures, intelligence does not reside in a single function.

It emerges from the system as a whole.

This systems perspective is not optional—it is essential. Without it, AI systems remain fragile, opaque, and difficult to improve. With it, engineers can build intelligent applications that are robust, understandable, and continuously evolving.

The next part of this book turns to the practical question: How do we create the environments and tools that enable this systems thinking? We begin by examining the workstation itself—not merely as a coding machine, but as an experimental platform for building intelligent systems.
