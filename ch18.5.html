22  Chapter 18.5 — Intent-Driven and Specification-Driven Systems

22.1 Introduction

In traditional command-line computing, the user writes explicit commands. Example:

git commit -m "update"
docker build .
kubectl apply -f deployment.yaml

Here the user specifies exact instructions. However, modern systems increasingly move toward a higher-level model where users express intent, and the system determines the execution steps.

Example:

Deploy this service
Analyze this dataset
Summarize these documents

Instead of commands, users provide goals. This leads to a new architecture:

Intent
   ↓
Specification
   ↓
Execution

This chapter explores this evolution from commands to intent, and how specification-driven systems transform AI engineering.

22.2 The Three-Layer Model

22.2.1 Intent Layer

The intent layer captures what the user wants to achieve. Examples:

build project
deploy service
summarize documents
search web

The intent layer is goal-oriented rather than instruction-oriented. Sources of intent may include: - Human input - API requests - AI prompts - System events

22.2.2 Specification Layer

The specification layer translates intent into structured plans. Example transformation:

Intent:
Deploy service

Specification:

build container
push image
update deployment
restart service

This layer defines: - Tasks - Dependencies - Parameters - Tools

Specifications are often expressed using structured formats: - YAML - JSON - DSL - Workflow graphs

22.2.3 Execution Layer

The execution layer performs the actual operations. Example execution flow:

task: build_image
tool: docker build

task: push_image
tool: docker push

task: deploy_service
tool: kubectl apply

The execution layer interacts with: - Programs - Services - Operating system - Infrastructure

22.3 Examples of Intent Systems

22.3.1 DevOps Systems

Example: Kubernetes.

Intent:

I want three replicas of this service.

Specification:

replicas: 3
image: myapp

Execution:

scheduler creates containers
network configured
service exposed

22.3.2 Build Systems

Example: Make.

Intent:

build project

Specification:

build:
    gcc main.c

Execution:

compile code
link binaries
produce executable

22.3.3 Workflow Systems

Example: Airflow.

Intent:

run data pipeline

Specification:

extract → transform → load

Execution:

task workers execute pipeline stages

22.3.4 Agent Systems

Intent:

find latest research papers

Specification:

search arXiv
filter papers
summarize results

Execution:

API calls
document processing
LLM summarization

22.4 The Interpreter Layer

22.4.1 Bridge Between Intent and Execution

Between intent and execution there is often an interpreter. Architecture:

Intent
   ↓
Interpreter
   ↓
Specification
   ↓
Execution

Examples:

Shell:

command interpreter

Build systems:

Make parser

Agents:

LLM planner

The interpreter determines how intent becomes structured actions.

22.4.2 AI as Interpreter

In modern AI systems, the interpreter is often an AI model:

User intent: "Deploy my app"
AI interpreter:
    - Understands context (current infrastructure)
    - Generates specification (deployment YAML)
    - Determines execution steps

This represents a fundamental shift: the system now interprets high-level intent and generates low-level specifications.

22.5 Specification-Driven Systems

22.5.1 From Code to Specifications

Traditional software engineering focuses on writing code:

write program
compile program
run program

But many modern systems increasingly rely on specifications instead. Examples: - Dockerfile - Kubernetes YAML - Terraform configuration - GitHub Actions - Workflow DAGs

These systems define what the system should do, rather than directly implementing how it should do it.

22.5.2 Specification vs Program

Traditional program:

instructions
control flow
algorithms

Specification:

structure
constraints
desired state

Comparison:

Program Specification
Step-by-step instructions Declarative description
How to do something What should exist
Procedural Declarative

22.5.3 The Spec-Driven Architecture

Spec-driven systems follow a general architecture:

Specification
      ↓
Interpreter
      ↓
Planner
      ↓
Execution Engine
      ↓
System State

Each component has a role.

22.5.3.1 Specification Layer

Defines system intent in structured form:

services
resources
dependencies
constraints

Formats include: - YAML - JSON - DSL - Schemas

22.5.3.2 Interpreter Layer

Reads the specification and converts it into an internal representation:

parse YAML
validate schema
construct system model

22.5.3.3 Planner Layer

Determines what actions must be taken to achieve the desired state:

desired state: 3 containers
current state: 1 container
plan: create 2 containers

This process is sometimes called reconciliation.

22.5.3.4 Execution Layer

Executes actions required by the plan:

create container
allocate network
attach storage

The system updates reality until it matches the specification.

22.5.4 Desired State Systems

Spec-driven systems often rely on desired state control loops:

Desired State
     ↓
Controller
     ↓
Actual State

Controller logic:

while actual != desired:
    reconcile()

Example Kubernetes loop:

desired: 3 pods
actual: 2 pods
action: create pod

22.6 The Evolution of Computing Interfaces

22.6.1 Increasing Abstraction

The evolution of computing interfaces can be seen as a progression:

commands
   ↓
scripts
   ↓
workflows
   ↓
agents
   ↓
intent-driven systems
   ↓
specification-driven systems

Each stage increases the level of abstraction. Users move further away from low-level instructions.

22.6.2 From Imperative to Declarative

Imperative (Commands):

Do A, then do B, then do C

Declarative (Specifications):

The system should be in state X

The shift from imperative to declarative represents a fundamental change in how we interact with computing systems.

22.7 Graph-Based Execution

22.7.1 DAG-Based Systems

Modern execution systems often represent tasks as graphs:

task A → task B → task C

Or more complex:

      A
     / \
    B   C
     \ /
      D

This structure is called a Directed Acyclic Graph (DAG). DAGs enable: - Parallel execution - Dependency management - Fault recovery

22.7.2 Application in AI Systems

In AI systems, DAGs are used for: - Multi-step reasoning - Tool coordination - Workflow orchestration

Example:

understand query
     ↓
retrieve documents
     ↓
analyze documents
     ↓
synthesize answer

22.8 Integration with Agent Systems

22.8.1 Agents as Specification Generators

Spec-driven systems connect naturally with agent systems:

Intent
   ↓
Agent
   ↓
Specification
   ↓
Execution System

Example:

User: deploy this service
Agent:
    generate deployment spec
System:
    execute deployment

The agent acts as a specification generator.

22.8.2 Hybrid Intelligence

Modern systems combine human intent with AI interpretation:

Human: provides high-level intent
AI: generates detailed specifications
System: executes the plan
Human: monitors and adjusts

This creates a collaborative intelligence model where humans and AI work together at different abstraction levels.

22.9 The Unified Model

22.9.1 The Complete Stack

We can unify everything discussed so far into a single model:

Intent
   ↓
Architecture
   ↓
Specification
   ↓
Plan
   ↓
Execution
   ↓
System State

Each layer refines the previous one. Example:

Intent:
    build web service

Architecture:
    frontend → API → database

Specification:
    container images, replicas

Execution:
    deploy containers

State:
    running system

22.9.2 Examples Across Domains

Shell:

command → execution

Workflow:

task graph → execution

Agent:

intent → tool calls

Spec-driven system:

desired state → reconciliation

All follow the same fundamental pattern: abstract intent is progressively refined into concrete execution.

22.10 Practical Implications

22.10.1 For AI Engineers

Understanding intent-driven and specification-driven systems has important implications:

Design for Intent: - Think in terms of goals, not commands - Design systems that can interpret high-level intent - Provide clear interfaces for expressing intent

Leverage Specifications: - Use declarative configurations where possible - Design systems that can generate specifications - Implement reconciliation loops for desired state

Layered Architecture: - Separate intent, specification, and execution layers - Each layer should be independently testable - Provide visibility into each layer for debugging

22.10.2 Benefits

Increased Abstraction: - Users work at higher levels of abstraction - Less need to understand implementation details - Faster iteration and experimentation

Better Automation: - Systems can generate execution plans automatically - Complex workflows become declarative - AI can assist in specification generation

Improved Reliability: - Declarative specifications are easier to verify - Desired state systems self-heal - Clear separation of concerns

22.11 Challenges and Considerations

22.11.1 Complexity Hiding

While higher abstraction is beneficial, it also hides complexity:

  • Debugging becomes harder when intent is far from execution
  • Understanding system behavior requires tracing through multiple layers
  • Performance optimization may require understanding all layers

22.11.2 Tool Support

Intent-driven systems require sophisticated tooling:

  • Interpreters that can understand natural language
  • Planners that can generate efficient execution plans
  • Observability tools that trace intent to execution

22.11.3 Education and Documentation

Users must understand: - How to express intent effectively - What specifications are generated - How execution proceeds

This requires new forms of documentation and education.

22.12 The Future Direction

22.12.1 AI-Enhanced Specification

Future systems will increasingly use AI to: - Interpret natural language intent - Generate optimal specifications - Adapt specifications based on context - Learn from execution patterns

22.12.2 Self-Improving Systems

Systems will become capable of: - Analyzing their own performance - Adjusting specifications automatically - Optimizing execution plans - Learning from user feedback

22.12.3 The Ultimate Goal

The evolution toward intent-driven systems represents a fundamental shift:

From:

programming systems

To:

designing systems

Instead of writing individual instructions, developers increasingly define: - Architectures - Constraints - Desired outcomes

Systems then generate and execute the necessary operations.

22.13 Conclusion

Intent-driven and specification-driven systems represent the future of computing interfaces. They enable users to work at higher levels of abstraction while maintaining the precision and control of command-line systems.

For AI engineers, understanding these systems is essential because:

  1. AI systems are inherently intent-driven: Users express goals, AI determines actions
  2. Specifications provide structure: They bridge the gap between intent and execution
  3. Layered architecture enables flexibility: Each layer can evolve independently

The command line introduced the power of composition through pipelines. Intent-driven systems extend this power by automating command generation. Specification-driven systems provide the structure to make this automation reliable and scalable.

As AI systems become more sophisticated, the line between command, specification, and intent will continue to blur. But the underlying principle—transforming abstract intent into concrete execution—will remain constant.

The future of AI engineering lies in mastering these layers and designing systems that can operate seamlessly across the entire abstraction stack.
